
## Определение

**Асимптотический анализ** — это метод оценки эффективности алгоритмов, который изучает, как изменяются затраты ресурсов (времени и памяти) при увеличении размера входных данных, без учёта констант и второстепенных факторов. Он позволяет сравнивать алгоритмы независимо от языка программирования, реализации и аппаратных характеристик.

---

## Предназначение асимптотического анализа

Если есть два алгоритма для одной задачи, как понять, какой из них лучше? Наивный способ — реализовать оба и запускать на разных входных данных, сравнивая время. У этого подхода есть проблемы:

- Результат может зависеть от конкретных входных данных.
- Разные компьютеры покажут разное время.

**Асимптотический анализ** решает эти проблемы. Мы оцениваем **порядок роста времени (или памяти)** алгоритма относительно размера входа, не измеряя фактическое время.

#### Пример:

Поиск элемента в **отсортированном массиве**:
- Линейный поиск — O(n)
- Бинарный поиск — O(log n)

Для маленьких n линейный поиск на быстром компьютере может быть быстрее. Но при больших n бинарный поиск **обязательно** станет быстрее, даже на медленном компьютере, из-за логарифмического роста.

## Анализ алгоритмов: худший, средний и лучший случаи

### 1. Худший случай (Worst Case) — используется чаще всего

Оценивается **верхняя граница** времени выполнения. Определяется сценарий, при котором алгоритм выполняет максимальное число операций.

*Пример:* линейный поиск — худший случай, когда элемент отсутствует (нужно проверить все n элементов).

**Почему популярен?**  
- Проще анализа среднего случая.  
- Даёт гарантию, что алгоритм не превысит определённое время.

### 2. Лучший случай (Best Case) — используется редко

Оценивается **нижняя граница**. Определяется сценарий с минимальным числом операций.

*Пример:* линейный поиск — лучший случай, когда элемент находится в первой позиции (O(1)).

Малоинформативен, так как не отражает реальную производительность.

### 3. Средний случай (Average Case) — используется редко

Учитываются все возможные входные данные, их распределение и время выполнения для каждого.

*Пример:* линейный поиск при равномерном распределении — в среднем потребуется проверить половину элементов (O(n)).

Сложен для точного расчёта, требует знания распределения входов.

## Сравнение Big O, Big Omega, Big Theta

| №   | **Big O**                                                                    | **Big Omega (Ω)**                                                                   | **Big Theta (Θ)**                                                                   |
| --- | ---------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| 1   | Похож на ≤. Скорость роста алгоритма **меньше или равна** заданному значению. | Похож на ≥. Скорость роста **больше или равна** заданному значению.                 | Похож на ==. Скорость роста **равна** заданному значению.                           |
| 2   | **Верхняя граница** функции.                                                 | **Нижняя граница** функции.                                                         | **Точная (жёсткая) граница** (и сверху, и снизу).                                   |
| 3   | Рассматривается сценарий максимального времени/памяти.                       | Рассматривается сценарий минимального времени/памяти.                               | Применим, если скорость роста **не меняется** с размером входа.                     |
| 4   | Математически: $0 ≤ f(n) ≤ C·g(n)$ для всех $n ≥ n₀$                         | Математически: $0 ≤ C·g(n) ≤ f(n)$ для всех $n ≥ n₀$                                | Математически: $0 ≤ C₂·g(n) ≤ f(n) ≤ C₁·g(n)$ для всех $n ≥ n₀$                     |

## Асимптотическая сложность (распространённые классы)

| Сложность       | Тип роста               | Пример алгоритма                                                          | Примечание                                       |
| --------------- | ----------------------- | ------------------------------------------------------------------------- | ------------------------------------------------ |
| $O(1)$          | Константный             | Доступ к элементу массива по индексу                                      | Время не зависит от размера входа                |
| $O(\log n)$     | Логарифмический         | Бинарный поиск                                                            | Очень эффективно при больших n                    |
| $O(n)$          | Линейный                | Линейный поиск, поиск минимума в массиве                                  | Растёт пропорционально n                          |
| $O(n \log n)$   | Линейно-логарифмический | Быстрая сортировка (в среднем), сортировка слиянием                       | Оптимально для сортировки больших данных          |
| $O(n^2)$        | Квадратичный            | Пузырьковая сортировка, вложенные циклы                                   | Приемлемо только для небольших n (≤ 10⁴)          |
| $O(n^3)$        | Кубический              | Тройной вложенный цикл, простое динамическое программирование на матрицах | Обычно для очень маленьких n (≤ 500)              |
| $O(2^n)$        | Экспоненциальный        | Перебор всех подмножеств, рекурсивный Фибоначчи без мемоизации            | Непрактично при n > 20                            |
| $O(n!)$         | Факториальный           | Полный перебор перестановок                                               | Только для n ≤ 10                                  |

## Связь с теоремой Мастера

Для рекурсивных алгоритмов типа **разделяй и властвуй** часто используется [[Теорема Мастера]]. Она позволяет быстро определить асимптотику для соотношений вида:

$$ T(n) = aT(n/b) + f(n) $$

где:
- $a$ — количество подзадач,
- $n/b$ — размер каждой подзадачи,
- $f(n)$ — работа вне рекурсивных вызовов.

Теорема даёт три случая для сравнения $f(n)$ с $n^{\log_b a}$ и определяет итоговый рост $T(n)$.

## Заключение

Асимптотический анализ — фундаментальный инструмент теории алгоритмов, позволяющий объективно сравнивать их эффективность. Он фокусируется на **порядке роста** затрат, игнорируя константы и детали реализации. На практике чаще всего используется анализ **худшего случая**, дающий гарантию производительности. Однако важно помнить об ограничениях: для малых входных данных или при больших константах асимптотически худший алгоритм может оказаться быстрее.

Понимание асимптотических обозначений (O, Ω, Θ) необходимо для профессиональной разработки и выбора оптимальных решений при работе с большими данными.
