## Определение

**Задача Best Time to Buy and Sell Stock III** — это классическая задача динамического программирования, в которой трейдер может совершить **не более двух транзакций** (покупка и продажа) для получения максимальной прибыли. Транзакция определяется как покупка акции и последующая продажа, причём нельзя владеть более чем одной акцией одновременно. Задача обобщает случай одной транзакции и является важным шагом к пониманию более сложных ограничений (k транзакций).

**Пример:**
```
prices = [3, 3, 5, 0, 0, 3, 1, 4]
Максимальная прибыль: 6 (купить по 0 в день 3, продать по 3 в день 5, купить по 1 в день 6, продать по 4 в день 7)
```

---

## Постановка задачи

**Вход:** массив `prices` (длина n), где `prices[i]` — цена акции в день i.
**Выход:** максимальная прибыль, которую можно получить, совершив **не более двух транзакций**. Транзакция — это покупка и продажа (в указанном порядке). Нельзя владеть более чем одной акцией одновременно.

**Ограничения:** типичные: 1 ≤ n ≤ 10⁵, 0 ≤ prices[i] ≤ 10⁴.

---

## Идея решения

Задача решается с помощью динамического программирования. Состояние можно описать тремя параметрами: день `i`, количество оставшихся транзакций `k` (0, 1, 2) и состояние `buy` (1 — можно покупать, 0 — нужно продавать). Рекуррентные соотношения:

- Если можно покупать (`buy = 1`):
  ```
  dp[i][k][1] = max(-prices[i] + dp[i+1][k][0], dp[i+1][k][1])
  ```
  (купить сегодня и перейти в состояние продажи, или пропустить день)

- Если нужно продавать (`buy = 0`):
  ```
  dp[i][k][0] = max(prices[i] + dp[i+1][k-1][1], dp[i+1][k][0])
  ```
  (продать сегодня, потратив одну транзакцию, или пропустить)

Базовые случаи: когда дней не осталось или `k == 0`, прибыль равна 0.

Ответ: `dp[0][2][1]`.

Мы рассмотрим два эффективных подхода, которые сокращают память и время.

---

## Подход 1: Bottom-Up DP с пространственной оптимизацией (O(n) времени, O(1) памяти)

### Идея

Из рекуррентных соотношений видно, что текущий день зависит только от значений следующего дня (`i+1`) и дня после следующего? В данном случае для `buy` используется `dp[i+1]`, а для `sell` используется `dp[i+1]` и `dp[i+2]`? В нашем случае для состояния `sell` при продаже мы переходим к `dp[i+1][k-1][1]`, что соответствует следующему дню с уменьшенным `k`. Никаких `i+2` здесь нет. Однако в общем случае для cooldown использовалось `i+2`, но здесь транзакции независимы, поэтому достаточно хранить только `dp[i+1]`.

Тем не менее, чтобы избежать полной 3D-таблицы, можно использовать два 2D-массива размера `(k+1) × 2` для текущего и следующего дней. Поскольку `k` ограничено 2, это константа.

Мы будем идти с конца к началу, обновляя `curr` (текущий день) на основе `next` (следующий день). Для каждого `k` от 1 до 2 вычисляем `curr[k][1]` и `curr[k][0]`. После завершения дня, копируем `curr` в `next`.

### Алгоритм

1. Инициализировать `next` — массив размером `(3) × 2`, заполненный нулями (для `k=0` всегда 0).
2. Для `i` от `n-1` до `0`:
   - Для `k` от 1 до 2:
     - `curr[k][1] = max(-prices[i] + next[k][0], next[k][1])`
     - `curr[k][0] = max(prices[i] + next[k-1][1], next[k][0])`
   - После вычислений присвоить `next = curr` (копия).
3. Вернуть `curr[2][1]` (или `next[2][1]` после цикла).

### Реализация

```python
def maxProfit(prices):
    n = len(prices)
    if n == 0:
        return 0
    # next содержит значения для дня i+1
    next_dp = [[0] * 2 for _ in range(3)]  # k=0,1,2; buy=0,1
    curr = [[0] * 2 for _ in range(3)]

    for i in range(n - 1, -1, -1):
        for k in range(1, 3):
            # buy state
            curr[k][1] = max(-prices[i] + next_dp[k][0], next_dp[k][1])
            # sell state
            curr[k][0] = max(prices[i] + next_dp[k-1][1], next_dp[k][0])
        # copy curr to next
        next_dp = [row[:] for row in curr]

    return curr[2][1]
```

### Сложность

- **Время:** O(n) — один проход по массиву.
- **Память:** O(1) — массивы размером 3×2, константа.

## Подход 2: Дальнейшая оптимизация с четырьмя переменными (O(n) времени, O(1) памяти)

### Идея

Можно ещё сильнее упростить, наблюдая, что состояния для двух транзакций могут быть выражены через четыре переменные, соответствующие максимальной прибыли после первой покупки, первой продажи, второй покупки и второй продажи.

- `firstBuy` — максимальная прибыль после первой покупки (обычно отрицательная, так как мы тратим деньги).
- `firstSell` — максимальная прибыль после первой продажи.
- `secondBuy` — максимальная прибыль после второй покупки (используя прибыль от первой продажи).
- `secondSell` — максимальная прибыль после второй продажи (итог).

Обновляем их последовательно для каждой цены:

```
firstBuy = max(firstBuy, -price)
firstSell = max(firstSell, firstBuy + price)
secondBuy = max(secondBuy, firstSell - price)
secondSell = max(secondSell, secondBuy + price)
```

Это моделирует две транзакции, используя тот факт, что покупка второй акции возможна только после продажи первой.

### Алгоритм

1. Инициализировать `firstBuy = -inf`, `firstSell = 0`, `secondBuy = -inf`, `secondSell = 0`.
2. Для каждой `price` в `prices`:
   - `firstBuy = max(firstBuy, -price)`
   - `firstSell = max(firstSell, firstBuy + price)`
   - `secondBuy = max(secondBuy, firstSell - price)`
   - `secondSell = max(secondSell, secondBuy + price)`
3. Вернуть `secondSell`.

### Реализация

```python
def maxProfit(prices):
    firstBuy = float('-inf')
    firstSell = 0
    secondBuy = float('-inf')
    secondSell = 0

    for price in prices:
        firstBuy = max(firstBuy, -price)
        firstSell = max(firstSell, firstBuy + price)
        secondBuy = max(secondBuy, firstSell - price)
        secondSell = max(secondSell, secondBuy + price)

    return secondSell
```

### Сложность

- **Время:** O(n)
- **Память:** O(1)

---

## Сравнение подходов

| Подход | Время | Память | Примечание |
|--------|-------|--------|------------|
| 3D DP (не показан) | O(n·k) | O(n·k) | базовый, но громоздкий |
| Bottom-Up с оптимизацией | O(n) | O(1) | использует два 2D-массива (константа) |
| Четыре переменные | O(n) | O(1) | самый простой и быстрый на практике |

---

## Примечания

- Оба ожидаемых подхода работают для любого количества транзакций, но для `k=2` они особенно компактны.
- Важно инициализировать `firstBuy` и `secondBuy` как `-inf`, чтобы корректно обработать первую покупку.
- В подходе с четырьмя переменными порядок обновлений важен: сначала `firstBuy`, затем `firstSell`, затем `secondBuy`, затем `secondSell`. Это гарантирует, что каждая транзакция использует правильные предыдущие значения.
- Если разрешено менее двух транзакций, ответ всё равно корректен, так как `secondSell` может остаться равным `firstSell` (если вторая транзакция невыгодна).
