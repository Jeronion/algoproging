## Определение

Даны два массива целых чисел `a` и `b` одинаковой длины. Суммарная комбинация образуется сложением одного элемента из `a` и одного из `b`. Каждая пара индексов `(i, j)` может быть использована не более одного раза. Требуется найти `k` **максимальных** сумм среди всех возможных комбинаций и вернуть их в порядке невозрастания.

**Пример:**
```
a = [3, 2], b = [1, 4], k = 2
Все суммы: 3+1=4, 3+4=7, 2+1=3, 2+4=6
Топ 2: [7, 6]
```

---

## Постановка задачи

**Вход:**
- `a` и `b` — массивы целых чисел (длина n ≤ 10⁵).
- `k` — количество максимальных сумм, которое нужно вернуть (1 ≤ k ≤ n).

**Выход:** массив из `k` наибольших сумм, отсортированных по убыванию.

---

## Идея решения (жадный алгоритм с max-heap)

Если просто сгенерировать все n² сумм и выбрать k наибольших, это потребует O(n²) памяти и времени, что неприемлемо при n до 10⁵.

Однако можно использовать следующий подход:

1. **Отсортировать оба массива по убыванию.** Тогда наибольшая сумма всегда будет `a[0] + b[0]`.
2. На каждом шаге мы будем извлекать текущую максимальную сумму из кучи, затем добавлять в кучу две новые комбинации:
   - `(i, j+1)` — следующий элемент из `b` с тем же `a[i]`.
   - `(i+1, j)` — следующий элемент из `a` с тем же `b[j]`.
3. Чтобы избежать дублирования пар, нужно гарантировать, что каждая пара `(i, j)` будет добавлена в кучу только один раз. Это достигается тем, что мы начинаем с `(0, 0)`, а при добавлении `(i, j+1)` делаем это только если `i == 0` (чтобы не дублировать), а `(i+1, j)` добавляется всегда, но только если `i+1 < n`. В коде реализовано: при извлечении пары `(i, j)`:
   - Если `i == 0` и `j + 1 < n`, то добавить `(0, j+1)`.
   - Если `i + 1 < n`, то добавить `(i+1, j)`.

Почему это гарантирует, что все топовые суммы будут найдены? Потому что массив отсортирован по убыванию, поэтому для любой пары `(i, j)`, её соседи `(i, j+1)` и `(i+1, j)` имеют суммы не больше (или равные, если равные значения). При извлечении максимальной суммы мы добавляем её «соседей», которые могут быть следующими по величине. Таким образом, мы перебираем все пары в порядке убывания суммы, не перебирая все n².

В куче хранятся кортежи `(-сумма, i, j)`, чтобы извлекать максимальную сумму.

---

## Алгоритм (пошагово)

1. Отсортировать `a` и `b` по убыванию.
2. Создать max-heap (min-heap с отрицательными значениями) и поместить в него `(-(a[0] + b[0]), 0, 0)`.
3. Инициализировать пустой список `res`.
4. Повторить `k` раз:
   - Извлечь из кучи `(neg_sum, i, j)`.
   - Добавить в `res` значение `a[i] + b[j]`.
   - Если `i == 0` и `j + 1 < len(b)`, добавить в кучу `(-(a[i] + b[j+1]), i, j+1)`.
   - Если `i + 1 < len(a)`, добавить в кучу `(-(a[i+1] + b[j]), i+1, j)`.
5. Вернуть `res`.

---

## Реализация на Python

```python
import heapq

class Solution:
    def topKSumPairs(self, a, b, k):
        a.sort(reverse=True)
        b.sort(reverse=True)
        n = len(a)
        # max-heap: отрицательная сумма, индексы
        heap = [(-(a[0] + b[0]), 0, 0)]
        res = []
        for _ in range(k):
            neg_sum, i, j = heapq.heappop(heap)
            res.append(a[i] + b[j])
            # добавляем (i, j+1) только для первой строки, чтобы избежать дубликатов
            if i == 0 and j + 1 < n:
                heapq.heappush(heap, (-(a[i] + b[j+1]), i, j+1))
            # добавляем (i+1, j) для всех
            if i + 1 < n:
                heapq.heappush(heap, (-(a[i+1] + b[j]), i+1, j))
        return res
```

## Сложность

- **Время:** O(n log n + k log k) — сортировка O(n log n), каждая из `k` операций извлечения и вставки в кучу размера O(k) занимает O(log k). На практике `k` может быть до n, поэтому O(n log n).
- **Память:** O(k) для хранения кучи (максимум O(k) элементов).

## Примечания

- Алгоритм использует свойство монотонности: суммы уменьшаются при увеличении индексов.
- Важно правильно обрабатывать дублирующиеся пары: условие `i == 0` для добавления `(0, j+1)` гарантирует, что каждая пара добавляется только один раз (через путь от `(0,0)` вниз по строке 0 и по столбцам).
- Если элементы массивов могут быть отрицательными, алгоритм всё равно работает (сортировка по убыванию, суммы могут быть отрицательными, но куча всё равно будет выдавать максимальные).
- Альтернативное решение: бинарный поиск по значению суммы + подсчёт количества пар ≥ x. Но кучный метод проще и эффективнее для данной задачи.
