
## Определение

**Коллизия** — ситуация, когда хеш-функция возвращает одинаковый индекс для двух или более разных ключей. Коллизии неизбежны из-за ограниченного размера хеш-таблицы. Для их разрешения существуют специальные методы, которые делятся на две основные категории:

1. **Раздельное связывание (Separate Chaining)**
2. **Открытая адресация (Open Addressing)**

---

## 1. Раздельное связывание (Separate Chaining)

Каждая ячейка хеш-таблицы является указателем на **связный список** (или другую структуру) записей, у которых хеш-значение совпадает. При вставке новый элемент добавляется в соответствующий список.

**Пример:**  
Хеш-функция: `key % 5`, ключи: 12, 15, 22, 25, 37.  

- 12 % 5 = 2 → список[2] = [12]
- 15 % 5 = 0 → список[0] = [15]
- 22 % 5 = 2 → список[2] = [12, 22]
- 25 % 5 = 0 → список[0] = [15, 25]
- 37 % 5 = 2 → список[2] = [12, 22, 37]

**Плюсы:** простота, неограниченное количество элементов (теоретически).  
**Минусы:** дополнительная память для указателей, возможное ухудшение производительности при длинных списках.

---

## 2. Открытая адресация (Open Addressing)

Все элементы хранятся **непосредственно в таблице**. При коллизии ищется другая свободная ячейка по определённому правилу (последовательность проб). Требуется, чтобы размер таблицы был больше количества хранимых элементов.

### 2.1 [[Линейное зондирование]] (Linear Probing)

При коллизии последовательно проверяются следующие ячейки:

```
index = (hash(key) + i) % size,   i = 0, 1, 2, ...
```

**Пример:**  
Хеш-функция: `key % 5`, ключи: 50, 70, 76, 85, 93. Таблица размера 5.

| Ключ | hash | i=0 | i=1 | i=2 |
|------|------|-----|-----|-----|
| 50   | 0    | 0   |     |     |
| 70   | 0    | занят → 1 | 1   |     |
| 76   | 1    | занят → 2 | 2   |     |
| 85   | 0    | занят → 1 занят → 2 занят → 3 | 3   |     |
| 93   | 3    | 3 занят → 4 | 4   |     |

**Плюсы:** простота. **Минусы:** первичная кластеризация (скопления занятых ячеек), что замедляет поиск.

### 2.2 Квадратичное зондирование (Quadratic Probing)

Индекс вычисляется по квадратичному закону, чтобы уменьшить кластеризацию:

```
index = (hash(key) + i²) % size,   i = 0, 1, 2, ...
```

**Пример:**  
Таблица размера 7, хеш-функция `x % 7`, ключи: 22, 30, 50.

- 22 % 7 = 1 → ячейка 1
- 30 % 7 = 2 → ячейка 2
- 50 % 7 = 1 → коллизия: (1 + 1²) % 7 = 2 (занято), (1 + 2²) % 7 = 5 → свободно.

**Плюсы:** уменьшает первичную кластеризацию. **Минусы:** возможна вторичная кластеризация, не всегда находит свободную ячейку даже если она есть.

### 2.3 [[Двойное хеширование]] (Double Hashing)

Используются две хеш-функции: `h1(key)` для начального индекса и `h2(key)` для шага. При коллизии применяется формула:

```
index = (h1(key) + i * h2(key)) % size,   i = 0, 1, 2, ...
```

Обычно `h2(key)` выбирают так, чтобы она была взаимно простой с размером таблицы.

**Пример:**  
Таблица размера 7, `h1(k) = k % 7`, `h2(k) = 1 + (k % 5)`. Ключи: 27, 43, 692, 72.

- 27: h1=6 → ячейка 6
- 43: h1=1 → ячейка 1
- 692: h1=6 → коллизия. h2=1+(692%5)=1+2=3. Новая позиция: (6+1*3) % 7 = 2 → свободно.
- 72: h1=2 → занято. h2=1+(72%5)=1+2=3. (2+1*3)%7=5 → свободно.

**Плюсы:** минимальная кластеризация, хорошее распределение. **Минусы:** более медленное вычисление.

## Сравнение методов

| Метод                     | Память         | Кластеризация       | Производительность (средняя) |
| ------------------------- | -------------- | ------------------- | ---------------------------- |
| Раздельное связывание     | O(n + m)       | нет (списки)        | O(1 + α) (α — load factor)   |
| Линейное зондирование     | O(m) (таблица) | первичная (высокая) | O(1) при низкой загрузке     |
| Квадратичное зондирование | O(m)           | вторичная (средняя) | O(1) при низкой загрузке     |
| Двойное хеширование       | O(m)           | очень низкая        | O(1) при низкой загрузке     |

*α = n/m (коэффициент заполнения)*

---

## Выбор метода

- **Раздельное связывание** — когда размер таблицы не критичен, а количество элементов может сильно варьироваться.
- **Открытая адресация** — когда экономия памяти важна (все данные хранятся в таблице), и коэффициент заполнения не превышает 0.7.
- **Двойное хеширование** — лучший выбор для открытой адресации, так как даёт наиболее равномерное распределение.
